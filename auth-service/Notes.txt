
why only @Required arg const in  service

 @RequiredArgsConstructor generates a constructor with required arguments (final fields or @NonNull).
 This is useful for dependency injection. Spring will automatically inject dependencies into this constructor.

 Why not use @NoArgsConstructor?
 A no-args constructor is not recommended when you have dependencies because it doesn't allow Spring
 to inject those dependencies automatically. You'd need to manually set the dependencies later.

 Why not use @Getter and @Setter?
 @Getter and @Setter are used to generate getters and setters for fields, but they are not needed
 when you’re using constructor-based injection, as Spring injects dependencies via the constructor.

The @ExceptionHandler(MethodArgumentNotValidException.class) annotation tells Spring that this method should be called whenever a MethodArgumentNotValidException is thrown. This exception is commonly thrown when the request body or parameters don't match the constraints defined in a Spring model (such as @NotNull, @Size, etc.).
2. Create the Error Response (Map<String, Object>):
Map<String, Object> errors = new HashMap<>();: A Map is used to structure the response, where the key is the field name and the value is the associated error message or other relevant data.
The errors map will contain metadata about the error, like the timestamp of the error, the HTTP status, and the actual validation errors.
3. Timestamp:
errors.put("timestamp", LocalDateTime.now().toString());: Adds a timestamp to the error response, so you know when the error occurred. The timestamp is the current date and time.
4. Status:
errors.put("status", HttpStatus.BAD_REQUEST.value());: Adds an HTTP status code (400 for bad request) to indicate that the request is invalid due to failed validation.
5. Field Errors:
Map<String, String> fieldErrors = new HashMap<>();: Creates a new map to hold specific validation errors related to the individual fields in the request.
ex.getBindingResult().getFieldErrors() gets all the field errors from the exception. Each field error contains details about which field was invalid and the associated error message.

The forEach loop iterates through the field errors and populates the fieldErrors map with field names as keys and error messages as values:

ex.getBindingResult().getFieldErrors()
    .forEach(e -> fieldErrors.put(e.getField(), e.getDefaultMessage()));

For example, if the request contains an invalid email, it would put something like:

fieldErrors.put("email", "Email should be valid.");
6. Add Field Errors to the Main Errors Map:
errors.put("errors", fieldErrors);: Adds the map of field-specific errors to the main errors map, under the key "errors". This makes the field-specific validation errors part of the overall error response.
7. Return the Response:
return ResponseEntity.badRequest().body(errors);: This line sends the error response back to the client with a 400 Bad Request HTTP status, along with the structured error details in the errors map.
Example Response:

If the client sends a request with invalid data (e.g., a missing email field or an invalid name field), the response might look like this:

{
  "timestamp": "2026-03-25T14:10:32.123",
  "status": 400,
  "errors": {
    "email": "Email should be valid.",
    "name": "Name cannot be empty."
  }
}
Key Points:
MethodArgumentNotValidException is thrown when a validation constraint is violated in a request.
The handler method collects all field errors from the exception and returns them in a structured JSON response.
The response contains metadata such as the timestamp, HTTP status code, and the specific validation errors.
Use Case:

This exception handler is useful for providing a clear, structured error response when a user submits data that doesn't conform to your model's validation constraints. Instead of just returning a generic error, you return detailed information about which fields were invalid and what the validation messages were.
